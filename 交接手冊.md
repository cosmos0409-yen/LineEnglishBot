# 英文導師 LINE 官方帳號 - 交接手冊

> 最後更新：2026-04-05

## 專案概述

本專案是一套「學生透過 LINE 發問 → AI 生成解答 → 雙導師審核 → 回覆學生」的自動化英文教學輔助系統。

### 系統架構總覽

```
學生 (LINE)                         導師 (Telegram + 審核網頁)
    │                                        │
    │  傳送文字/圖片                           │
    ▼                                        │
LINE Messaging API                           │
    │                                        │
    ▼                                        │
┌───────────────────────────────┐             │
│  FastAPI 後端 (Railway)        │             │
│                               │             │
│  POST /webhook                │             │
│    ├─ 驗證 LINE 簽章            │             │
│    └─ BackgroundTasks ────────┤             │
│         │                     │             │
│         ├─ 1. 下載學生圖片     │              │
│         ├─ 2. GPT-5.4-mini 解答 │             │

│         └─ 4. Telegram 推播 ──┤────────────▶│ 收到通知 + 審核連結
│                               │             │
│  GET  /review/{id}  ◀─────────┤─────────────│ 點擊連結進入審核

│    └─ 更新 DB 審核狀態         │              │
```

### 技術堆疊

| 類別 | 技術 | 用途 |
|------|------|------|
| 後端框架 | FastAPI (Python) | 非同步 Web 框架，處理 Webhook 與審核路由 |
| 學生端 | LINE Messaging API | 學生透過 LINE 發問與接收回覆 |
| AI 引擎 | OpenAI GPT-5.4-mini | 圖片 OCR + 英文解答生成 |
| 導師通知 | Telegram Bot API | 免費無限推播，通知導師審核 |
| 審核介面 | FastAPI + Jinja2 + HTML | 輕量網頁，供導師編輯/批准解答 |
| 資料庫 | Supabase (PostgreSQL) | 儲存問題、解答、審核狀態 |
| 部署 | Railway | 無休眠的免費/低成本雲端主機 |

> **注意**：原本規劃使用 Render Free Tier，但 Render 在 2024 年底更新政策，會主動偵測並忽略 keep-alive ping，Free Tier 服務仍會休眠，導致 LINE Webhook 逾時失敗。改用 **Railway** 可避免此問題。

## 各檔案詳細職責

### `main.py` — 主程式入口

| 路由 | 方法 | 說明 |
|------|------|------|
| `/` | GET | 健康檢查端點，回傳伺服器運行狀態 |
| `/webhook` | POST | 接收 LINE Webhook 事件，驗證簽章後丟進背景任務處理 |
| `/review/{ticket_id}` | GET | 顯示審核頁面（學生提問 + AI 解答 + 編輯區） |
| `/review/{ticket_id}` | POST | 處理導師的審核提交（更新解答 + 遞增審核計數 + 觸發 LINE Push） |

**核心流程（10 秒訊息緩衝）**：

`buffer_event(event)` → 將訊息加入緩衝，重新計時 10 秒

`_delayed_process(user_id)` → 10 秒後觸發，取出所有緩衝訊息

`_process_batch(user_id, events)` → 批次處理：
1. 取得學生 LINE 顯示名稱（一次）
2. 下載所有圖片（取第一張）、合併所有文字
3. 呼叫 GPT-5.4-mini 產生 AI 解答
4. 存入 Supabase 資料庫
5. 推播到導師 Telegram 群組（含審核頁面連結）

> 同一學生在 10 秒內傳送的文字與圖片會合併成一筆處理，解決前後文關聯問題。

**審核流程 `review_submit()`**：
1. 防護：status 已為 `approved` 或 `sent` 時直接回傳，不重複處理
2. 呼叫 `supabase_service.approve_question()` 遞增 approval_count
3. 若 approval_count >= 2 → 呼叫 `line_service.push_message()` 推播給學生
4. Push 成功 → 呼叫 `supabase_service.mark_as_sent()` 將 status 更新為 `sent`

**簽章驗證**：透過環境變數 `DEV_MODE=true` 可跳過（僅限開發環境）。

### `services/line_service.py` — LINE API 服務

| 函式 | 說明 |
|------|------|
| `get_user_profile(user_id)` | 呼叫 LINE API 取得用戶顯示名稱 |

### `services/gemini_service.py` — AI 解答服務（現為 OpenAI 實作）

| 函式 | 說明 |
|------|------|
| `generate_answer(image_bytes, user_text)` | 呼叫 GPT-5.4-mini，支援純文字、純圖片、圖文混合三種情境 |

> 檔名保留 `gemini_service.py` 以維持 import 相容性，實際使用 OpenAI API。

**System Prompt 設計原則**：
- 角色定位：學生的英文老師，透過 LINE 幫學生解題，語氣自然像真人打字回覆
- 直接給答案，不加開場白或稱謂
- 格式嚴格禁止 Markdown（不用 `**`、`*`、`#`、`-` 等符號）
- 條列用數字加點（1. 2. 3.）或換行分段
- 每行結尾不加句號，不加總結收尾語
- 長度以「剛好夠懂」為原則

**延遲初始化**：`genai.Client` 使用 lazy init 模式（`_get_client()`），避免模組 import 時因 dotenv 尚未載入而報錯。

---

### `services/telegram_service.py` — Telegram 推播服務

| 函式 | 說明 |
|------|------|
| `send_text_message(text)` | 發送純文字訊息到導師群組 |
| `send_photo(image_bytes, caption)` | 發送圖片到導師群組（caption 上限 1024 字） |
| `notify_new_question(...)` | 整合推播：學生資訊 + 圖片 + AI 解答 + 審核連結 |

**訊息長度處理**：Telegram 單則上限 4096 字，超過會自動拆分發送。

---

### `services/supabase_service.py` — 資料庫服務

| 函式 | 說明 |
|------|------|
| `save_question(...)` | 新增一筆提問記錄（含 AI 解答） |
| `get_question(question_id)` | 根據 UUID 取得單筆提問 |
| `approve_question(question_id, edited_answer)` | 導師審核：更新解答、遞增 approval_count、自動判定 status |
| `mark_as_sent(question_id)` | LINE Push 成功後將 status 更新為 `sent`，防止重複發送 |

**技術細節**：直接使用 `httpx` 呼叫 Supabase PostgREST API，無需額外安裝 `supabase` 套件。所有操作皆為非同步。

---

### `templates/review.html` — 審核網頁介面

- **深色主題**介面，適合長時間閱讀
- 顯示：學生名稱、提問時間、圖片（如有）、AI 解答
- 可編輯的 textarea 讓導師修改解答內容
- 審核進度顯示（0/2、1/2、2/2）
- status 對應顯示：`pending` 等待審核、`approved` 已通過、`sent` 已發送給學生

### `setup_database.sql` — 資料庫結構

```sql
questions 資料表:
  id                    UUID (PK, auto-gen)
  student_line_id       TEXT        -- 學生 LINE User ID
  student_name          TEXT        -- 學生顯示名稱
  question_type         TEXT        -- 'text' 或 'image'
  question_text         TEXT        -- 文字提問內容（nullable）
  question_image_base64 TEXT        -- 圖片 base64 編碼（nullable）
  ai_answer             TEXT        -- AI 初版解答
  final_answer          TEXT        -- 最終解答（導師可修改）
  approval_count        INTEGER     -- 0 → 1 → 2（雙導師通過）
  status                TEXT        -- pending → approved → sent
  created_at            TIMESTAMPTZ -- 建立時間
```

## 開發環境建置教學

### 前置條件
- Python 3.10+
- LINE Official Account（Messaging API Channel）
- Telegram Bot + 導師群組
- Supabase 帳號 + 專案
- Git（建議）

### 1. 安裝套件

```bash
python -m venv venv
venv\Scripts\activate   # Windows
pip install -r requirements.txt
```

### 2. 環境變數設定

```bash
copy .env.example .env
# 填入所有金鑰
```

### 3. 建立資料庫

前往 Supabase Dashboard → SQL Editor，貼上 `setup_database.sql` 的內容並執行。

### 4. 啟動伺服器

```bash
python -m uvicorn main:app --reload --port 8000
```

### 5. LINE Webhook 設定

```bash
ngrok http 8000
# 將產生的 HTTPS 網址 + /webhook 填入 LINE Developers Console
```

## 部署到 Railway

1. 前往 [railway.app](https://railway.app) → New Project → Deploy from GitHub Repo
2. 選擇此 repository，Railway 會自動偵測 `railway.toml` 並使用 `Procfile` 的啟動指令
3. 專案設定 → Variables → 逐一填入所有環境變數
4. 部署完成後複製 Public Domain（如 `https://xxx.railway.app`）
5. 更新環境變數 `APP_BASE_URL = https://xxx.railway.app`（需 Redeploy 生效）
6. 將 `https://xxx.railway.app/webhook` 填入 LINE Developers Console → Webhook URL

## LINE Developers Console 設定指引

1. 前往 [LINE Developers Console](https://developers.line.biz/console/)
2. 建立 Provider → 新增 Messaging API Channel
3. **Basic settings**：取得 `Channel Secret`
4. **Messaging API**：
   - Issue `Channel Access Token (long-lived)`
   - 填入 Webhook URL（Railway URL + `/webhook`）
   - 開啟 **Use webhook**
   - 建議關閉 **Auto-reply messages**（LINE Official Account Manager 中設定）

## ⚠️ 已知限制與 TODO

1. **LINE 端對端尚未測試** — 缺少 LINE Channel Secret / Access Token，圖片下載與 Push 功能尚未實際驗證
2. **審核無身分識別** — 審核頁面不區分是哪位導師操作，同一人可以連續點兩次通過。可考慮加入簡易身分驗證（如導師名字選單）
3. **圖片以 base64 存進 DB** — 可用但不夠高效，大量使用後可改用 Supabase Storage
4. **日誌系統** — 目前使用 `print()` 輸出，正式環境應改為 `logging` 模組
